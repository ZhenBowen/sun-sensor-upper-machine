# 太阳敏上位机软件方案

## 1. 目标与边界

本文档用于定义太阳敏感器工程测试型上位机的软件方案。该上位机服务于太阳敏硬件联调、串口通信验证、遥测显示、数据记录、标定采集和测试报告数据整理。

当前 STM32 固件尚未完成，因此本文档同时给出推荐通信协议，供后续固件工程师和上位机开发对齐。最终协议以固件实现和双方确认的 ICD 为准。

本方案不直接复用电源模块上位机的 CAN 通信代码。电源模块上位机采用 `USB-CAN + ControlCAN.dll`，太阳敏感器任务书要求 `UART 115200 bps`，硬件上通过 THVD1400 转为 RS-485 差分物理层。电脑侧建议使用 `USB-RS485` 转换器，软件侧按串口 `COMx` 访问。

## 2. 设计目标

上位机应优先满足以下工程测试需求：

1. 稳定连接 USB-RS485 串口，支持 `115200, 8N1`。
2. 实时接收并解析太阳敏遥测帧。
3. 显示太阳入射角 `alpha`、`beta`，太阳存在标志、温度、状态字、帧率和丢帧统计。
4. 显示并记录四路原始 ADC：`VAX1`、`VAX2`、`VAY1`、`VAY2`。
5. 将每帧遥测保存为 CSV，用于标定、精度统计和测试报告。
6. 支持输入转台参考角 `alpha_ref`、`beta_ref`、测试点编号和备注，并随 CSV 一起记录。
7. 预留遥控命令接口，用于后续查询遥测、设置输出频率、设置节点 ID、进入校准模式等。

第一版不做复杂标定拟合和报告自动生成，只保证“收得到、看得见、存得下、能联调”。

## 5. 总体数据流

```text
太阳敏 STM32
  -> UART1_TX/RX
  -> THVD1400 RS-485
  -> RS485_A / RS485_B
  -> USB-RS485 转换器
  -> Windows COM 口
  -> sun_host 串口接收
  -> sun_protocol 帧解析与 CRC 校验
  -> SunTelemetry 数据对象
  -> sun_gui 状态刷新
  -> sun_monitor 曲线与表格
  -> sun_logger CSV 记录
```

上位机接收线程不应直接更新 PyQt 界面。推荐做法是接收线程解析出 `SunTelemetry` 后，通过 Qt signal 发送给主线程，由主线程统一更新界面和日志。

## 6. 串口通信方案

### 6.1 物理连接

太阳敏外部连接器输出 `RS485_A` 和 `RS485_B`，电脑侧使用 `USB-RS485` 转换器。连接时需要确认：

| 信号 | 说明 |
|---|---|
| `RS485_A` | 接 USB-RS485 的 A |
| `RS485_B` | 接 USB-RS485 的 B |
| `GND` | 建议与 USB-RS485 信号地共地，减少通信不稳定 |
| `VDDIN` | 外部供电，默认测试条件建议 5 V |

如果收不到数据，优先检查 A/B 是否接反、供电是否正常、STM32 是否真的在发送、USB-RS485 驱动是否生成 COM 口。

### 6.2 串口参数

按任务书默认：

| 参数 | 值 |
|---|---|
| 波特率 | `115200` |
| 数据位 | `8` |
| 校验 | `None` |
| 停止位 | `1` |
| 流控 | `None` |
| 工作模式 | 遥测 + 遥控 |

### 6.3 收发模式

THVD1400 是半双工 RS-485 收发器，STM32 侧需要用 `DE/RE#` 控制方向。建议固件策略：

1. 默认保持接收状态。
2. 需要发送遥测或应答时拉高 `DE`。
3. UART 发送完成后立即拉低 `DE`。

电脑侧 USB-RS485 通常自动处理发送方向，上位机只按普通串口读写即可。

## 7. 推荐遥测协议

由于 STM32 固件尚未定稿，建议采用二进制定长遥测帧。二进制帧比文本帧更稳定，适合后续高频记录和 CRC 校验。

### 7.1 字节序与缩放

| 项目 | 约定 |
|---|---|
| 字节序 | 小端序 Little Endian |
| 角度 | `int16`，单位 `0.01 deg` |
| 温度 | `int16`，单位 `0.01 degC` |
| ADC | `uint16`，原始 ADC 码值 |
| 时间戳 | `uint32`，单位 `ms` |
| CRC | `CRC-16/Modbus`，覆盖帧头之后、CRC 之前的所有字节 |

### 7.2 遥测帧格式

建议帧头使用 `0x55 0xAA`，避免与文本换行混淆。

| 偏移 | 长度 | 类型 | 字段 | 说明 |
|---:|---:|---|---|---|
| 0 | 2 | `uint16` | `sof` | 固定 `0xAA55`，线上字节为 `55 AA` |
| 2 | 1 | `uint8` | `version` | 协议版本，第一版为 `1` |
| 3 | 1 | `uint8` | `msg_type` | `0x01` 表示遥测 |
| 4 | 1 | `uint8` | `node_id` | 太阳敏节点 ID |
| 5 | 1 | `uint8` | `payload_len` | 从 `seq` 到 `status_word` 的长度，建议 `28` |
| 6 | 2 | `uint16` | `seq` | 帧序号，循环递增 |
| 8 | 4 | `uint32` | `timestamp_ms` | STM32 时间戳 |
| 12 | 2 | `uint16` | `adc_vax1` | PA0 原始 ADC |
| 14 | 2 | `uint16` | `adc_vax2` | PA1 原始 ADC |
| 16 | 2 | `uint16` | `adc_vay1` | PA2 原始 ADC |
| 18 | 2 | `uint16` | `adc_vay2` | PA3 原始 ADC |
| 20 | 2 | `int16` | `alpha_cdeg` | `alpha_deg * 100` |
| 22 | 2 | `int16` | `beta_cdeg` | `beta_deg * 100` |
| 24 | 2 | `int16` | `temp_centi_c` | `temp_C * 100` |
| 26 | 1 | `uint8` | `sun_present` | `0` 无太阳，`1` 检测到太阳 |
| 27 | 1 | `uint8` | `saturation_flag` | `0` 未饱和，`1` 饱和 |
| 28 | 2 | `uint16` | `status_word` | 状态位 |
| 30 | 2 | `uint16` | `crc16` | CRC-16/Modbus |

总帧长建议为 `32` 字节。

### 7.3 状态字定义

建议 `status_word` 按位定义：

| 位 | 名称 | 含义 |
|---:|---|---|
| 0 | `VALID` | 当前角度有效 |
| 1 | `SUN_PRESENT` | 检测到太阳 |
| 2 | `SATURATED` | 至少一路 ADC 饱和 |
| 3 | `LOW_SIGNAL` | 光强低于有效阈值 |
| 4 | `ADC_ERROR` | ADC 采样异常 |
| 5 | `TEMP_WARNING` | 温度超出建议范围 |
| 6 | `CALIBRATION_MODE` | 当前处于校准/调试模式 |
| 7 | `PARAM_ERROR` | 标定参数无效或未加载 |
| 8-15 | 保留 | 后续扩展 |

上位机显示时应同时显示原始十六进制状态字和按位解释结果。

## 8. 推荐遥控协议

第一版可以只接收周期遥测，不做复杂遥控。但协议层应预留遥控命令，方便后续联调。

遥控帧建议格式：

| 偏移 | 长度 | 类型 | 字段 | 说明 |
|---:|---:|---|---|---|
| 0 | 2 | `uint16` | `sof` | 固定 `0x55AA`，线上字节为 `AA 55` 或另选命令帧头 |
| 2 | 1 | `uint8` | `version` | 协议版本 |
| 3 | 1 | `uint8` | `msg_type` | `0x80` 表示遥控 |
| 4 | 1 | `uint8` | `node_id` | 目标节点 |
| 5 | 1 | `uint8` | `cmd_id` | 命令 ID |
| 6 | 1 | `uint8` | `payload_len` | 命令参数长度 |
| 7 | N | `bytes` | `payload` | 命令参数 |
| 7+N | 2 | `uint16` | `crc16` | CRC-16/Modbus |

推荐命令：

| 命令 ID | 名称 | 参数 | 说明 |
|---:|---|---|---|
| `0x01` | `QUERY_TELEMETRY` | 无 | 查询一次遥测 |
| `0x02` | `SET_OUTPUT_RATE` | `uint16 rate_hz` | 设置遥测输出频率 |
| `0x03` | `SET_NODE_ID` | `uint8 node_id` | 设置节点 ID |
| `0x04` | `ENTER_CALIB_MODE` | `uint8 enable` | 进入或退出校准模式 |
| `0x05` | `ZERO_DARK_OFFSET` | 无 | 采集暗场偏置 |
| `0x06` | `RESET_DEVICE` | 无 | 软件复位 |

## 10. CSV 记录方案

CSV 是后续标定和测试报告的核心。建议每帧写入一行。

推荐字段：

```csv
pc_time_iso,pc_time_ms,seq,node_id,timestamp_ms,
adc_vax1,adc_vax2,adc_vay1,adc_vay2,
alpha_deg,beta_deg,temp_c,
sun_present,saturation_flag,status_word,valid_flag,
frame_rate_hz,drop_count,crc_error_count,
alpha_ref_deg,beta_ref_deg,test_point,comment
```

字段说明：

| 字段 | 说明 |
|---|---|
| `pc_time_iso` | 电脑时间，便于和测试记录对应 |
| `pc_time_ms` | 电脑毫秒时间戳 |
| `timestamp_ms` | STM32 时间戳 |
| `seq` | 下位机帧序号 |
| `adc_*` | 四路原始 ADC |
| `alpha_deg` / `beta_deg` | 下位机解算角 |
| `alpha_ref_deg` / `beta_ref_deg` | 转台参考角 |
| `status_word` | 原始状态字 |
| `valid_flag` | 上位机从状态字解析出的有效标志 |
| `drop_count` | 上位机累计丢帧数 |
| `crc_error_count` | CRC 错误累计数 |

文件名建议：

```text
sun_sensor_YYYYMMDD_HHMMSS.csv
```

示例：

```text
sun_sensor_20260601_153000.csv
```

## 11. 协议解析与异常处理

上位机串口接收应使用字节流解析器，而不是假设每次 `read()` 都刚好读到一整帧。

解析流程：

1. 将串口读到的字节追加到缓冲区。
2. 在缓冲区中查找帧头 `55 AA`。
3. 帧头前的无效字节丢弃。
4. 如果缓冲区不足一整帧，等待下一次接收。
5. 读取 `payload_len` 并计算完整帧长。
6. 校验 CRC。
7. CRC 正确则生成 `SunTelemetry`。
8. CRC 错误则累计错误数，并从下一个字节继续搜索帧头。

异常处理：

| 异常 | 上位机处理 |
|---|---|
| 串口打不开 | 弹窗提示并保持未连接状态 |
| 设备断开 | 停止接收线程，状态栏提示断开 |
| CRC 错误 | 不更新实时值，只累计错误计数 |
| 帧序号跳变 | 计算丢帧数 |
| 长时间无数据 | 状态显示超时，曲线不再新增 |
| CSV 写入失败 | 停止记录并提示保存路径问题 |

## 12. 与 STM32 固件工程师的接口清单

需要尽早和固件工程师确认：

1. 是否采用本文推荐二进制帧。
2. 遥测是否周期主动上报，目标频率建议 `10 Hz`。
3. 是否需要上位机发送查询命令后才返回遥测。
4. 四路 ADC 是原始码值、滤波后码值，还是电压毫伏值。
5. `alpha`、`beta` 的正方向定义。
6. `AX1/AX2/AY1/AY2` 与物理方向的对应关系。
7. 角度和温度的缩放系数。
8. CRC 算法是否使用 `CRC-16/Modbus`。
9. 状态字每一位的定义。
10. 节点 ID 是否需要可配置。

在固件未完成前，上位机可以先实现模拟数据源，用于验证界面、曲线和 CSV 记录。

## 13. 开发阶段划分

### 阶段 1：通信与日志闭环

目标：串口能打开，能解析模拟帧或真实帧，能保存 CSV。

内容：

1. 实现 `sun_protocol.py` 的帧解析和 CRC。
2. 实现 `sun_host.py` 的串口连接和接收线程。
3. 实现 `sun_logger.py` 的 CSV 写入。
4. 做一个模拟数据源，用于在没有 STM32 时测试上位机。

验收标准：

1. 使用模拟数据源时，界面能连续更新。
2. CSV 行数随遥测帧增加。
3. 故意破坏 CRC 时，上位机能统计错误而不崩溃。

### 阶段 2：工程测试界面

目标：实时显示角度、ADC、状态和曲线。

内容：

1. 主界面连接控制区。
2. 实时遥测数值区。
3. 四路 ADC 曲线。
4. `alpha`、`beta` 曲线。
5. 丢帧率、帧率、CRC 错误显示。

验收标准：

1. `10 Hz` 模拟数据下界面不卡顿。
2. 曲线显示最近数据点。
3. 断开串口后界面状态正确。

### 阶段 3：标定采集辅助

目标：支持太阳模拟器和转台测试。

内容：

1. 输入 `alpha_ref_deg`、`beta_ref_deg`。
2. 输入测试点编号和备注。
3. CSV 每帧写入参考角。
4. 支持一键开始/停止当前测试点记录。

验收标准：

1. 每个测试点的数据可在 CSV 中按 `test_point` 区分。
2. 参考角不会因为界面刷新丢失。
3. 后处理脚本可以直接按测试点分组。

### 阶段 4：后处理与报告数据

目标：从 CSV 生成测试报告需要的统计结果。

内容：

1. 读取 CSV。
2. 按测试点计算均值、标准差。
3. 计算单点角误差。
4. 统计 `±60°`、`±75°`、`±80°` 范围 RMS。
5. 导出误差表格和基础图表。

验收标准：

1. 能输出任务书要求的 RMS 统计。
2. 能生成全视场误差表。
3. 能辅助填写 `10.4 校准与测试报告`。

### 阶段 5：遥控命令扩展

目标：上位机可主动控制太阳敏工作模式。

内容：

1. 查询遥测。
2. 设置输出频率。
3. 设置节点 ID。
4. 进入/退出校准模式。
5. 触发暗场偏置采集。

验收标准：

1. 每条命令有发送记录。
2. 下位机有明确 ACK 或状态变化。
3. 错误命令不会导致上位机卡死。

## 15. 测试建议

在真实硬件到位前，建议先做上位机自测：

1. 用模拟数据源生成 `10 Hz` 遥测帧。
2. 用协议单元测试验证 CRC、帧头搜索、断帧拼接。
3. 用 CSV 写入测试确认字段完整。
4. 用错误帧测试 CRC 错误统计。
5. 用帧序号跳变测试丢帧统计。

真实硬件到位后，按以下顺序联调：

1. 确认 USB-RS485 生成 COM 口。
2. 用串口调试助手确认是否有数据。
3. 用上位机连接并观察 CRC 错误计数。
4. 确认四路 ADC 在暗场和强光下变化合理。
5. 确认遮光时 `sun_present=0`。
6. 确认照光时 `sun_present=1`。
7. 在 `0°`、`+10° X`、`-10° X`、`+10° Y`、`-10° Y` 下确认角度正负方向。

## 16. 关键风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 固件协议未定 | 上位机无法最终联调 | 先实现推荐协议和模拟数据源，再按 ICD 调整 |
| A/B 接反 | 收不到或乱码 | 联调时准备反接测试 |
| DE 控制错误 | RS-485 总线无法正常收发 | 固件默认接收，仅发送时短暂拉高 DE |
| 只输出角度不输出 ADC | 无法做可靠标定和误差追因 | 要求固件保留四路原始 ADC |
| 文本协议字段漂移 | CSV 数据不稳定 | 正式版采用二进制帧 |
| 上位机主线程读串口 | 界面卡顿 | 串口接收放在线程中，通过 signal 更新 UI |

## 18. 结论

太阳敏上位机第一阶段应定位为工程测试工具，而不是展示型软件。核心价值是稳定获取数据、完整保存原始 ADC 和解算角度、辅助标定测试。只要上位机能可靠记录 `ADC + angle + status + timestamp + reference angle`，后续精度分析、RMS 统计和测试报告都可以建立在同一套数据上。

## 19. 详细需求分解

### 19.1 使用场景

上位机至少覆盖四类实际使用场景。

第一类是硬件通电联调。测试人员给太阳敏供电后，将 `RS485_A/B` 通过 USB-RS485 转换器接入电脑。上位机扫描串口、连接串口、显示是否收到数据、显示 CRC 是否正确、显示帧率是否达到预期。这个阶段的核心问题是确认“通信链路是否通”。

第二类是光电链路检查。测试人员在暗场、弱光、强光、遮挡、太阳模拟器照射等条件下观察四路 ADC 是否合理变化。这个阶段的核心问题是确认“PSD、模拟前端、ADC 和固件采样是否正常”。

第三类是角度解算检查。测试人员把太阳敏装到转台上，在 `0 deg`、`+10 deg`、`-10 deg`、`+30 deg`、`-30 deg` 等点位观察 `alpha`、`beta` 的正负方向和数值趋势。这个阶段的核心问题是确认“坐标系、符号、角度输出是否可信”。

第四类是标定与测试报告采集。测试人员在太阳模拟器和转台上按点位扫描，输入参考角和测试点编号，上位机把每一帧原始 ADC、角度、状态、参考角写入 CSV。这个阶段的核心问题是留下可以复算、可追溯、可写入报告的数据。

### 19.2 第一版必须完成的用户动作

第一版软件应让测试人员完成以下动作：

1. 打开软件后看到可用串口列表。
2. 选择串口并点击连接。
3. 看到连接状态、接收状态、帧率、CRC 错误数。
4. 看到当前 `alpha`、`beta`、温度、太阳存在状态。
5. 看到四路 ADC 当前值。
6. 看到四路 ADC 曲线和角度曲线。
7. 输入当前转台参考角和测试点编号。
8. 点击开始记录，选择或自动生成 CSV 文件。
9. 点击停止记录，CSV 正常关闭。
10. 串口断开、数据超时或 CRC 错误时能明确看到提示。

第一版不要求完成自动标定拟合、自动控制转台、自动生成完整 Word/PDF 报告。

### 19.3 成功标准

第一版软件完成后，建议用以下标准验收：

| 项目 | 验收标准 |
|---|---|
| 串口连接 | 能打开和关闭 USB-RS485 对应 COM 口 |
| 数据接收 | `10 Hz` 模拟数据连续运行 `30 min` 不崩溃 |
| 协议解析 | 支持断帧、粘包、无效字节、CRC 错误恢复 |
| 显示刷新 | 实时数值和曲线刷新不卡顿 |
| 数据记录 | CSV 字段完整，帧数与接收帧数一致 |
| 错误统计 | CRC 错误、丢帧、超时均有计数 |
| 标定辅助 | 每帧 CSV 能写入参考角和测试点编号 |

### 20.2 `sun_protocol.py`

该文件负责“字节”和“对象”之间的转换。它不直接读串口，也不更新界面。

建议包含：

| 名称 | 类型 | 职责 |
|---|---|---|
| `SOF = b"\x55\xAA"` | 常量 | 遥测帧头 |
| `MSG_TELEMETRY = 0x01` | 常量 | 遥测消息类型 |
| `MSG_COMMAND = 0x80` | 常量 | 遥控消息类型 |
| `crc16_modbus(data)` | 函数 | 计算 CRC |
| `parse_telemetry_frame(frame)` | 函数 | 单帧解析为 `SunTelemetry` |
| `build_command_frame(...)` | 函数 | 打包遥控命令 |
| `TelemetryParser` | 类 | 字节流解析器 |

`TelemetryParser` 内部维护一个 `bytearray` 缓冲区。串口每读到一段字节，就调用：

```python
frames = parser.feed(data)
```

返回值是解析成功的 `SunTelemetry` 列表。这样可以处理以下情况：

| 输入情况 | 处理方式 |
|---|---|
| 一次读到半帧 | 缓存在内部，等待下一次 |
| 一次读到多帧 | 循环解析，返回多个对象 |
| 帧前有乱码 | 丢弃帧头前无效字节 |
| CRC 错误 | 错误计数加一，从下一个候选帧头继续 |
| payload 长度异常 | 丢弃当前帧头并继续搜索 |

状态机可以按以下逻辑实现：

```text
WAIT_SOF
  找到 55 AA 后进入 READ_HEADER

READ_HEADER
  确认至少已有 version/msg_type/node_id/payload_len
  根据 payload_len 计算完整帧长

READ_BODY
  缓冲区不足完整帧时等待
  缓冲区足够时截出候选帧

VERIFY_CRC
  CRC 正确则解析对象
  CRC 错误则统计错误并滑动一个字节继续搜索

EMIT
  返回 SunTelemetry 给上层
```

建议解析时严格检查：

1. `sof` 必须正确。
2. `version` 必须是支持版本。
3. `msg_type` 必须是 `0x01`。
4. `payload_len` 必须等于预期长度，第一版建议固定 `24` 或 `28`，以最终协议为准。
5. CRC 必须正确。
6. 帧长必须与 `payload_len` 一致。

### 20.3 `sun_host.py`

该文件负责串口生命周期。它不关心界面布局，也不直接写 CSV。

建议设计一个 `SunSerialHost`：

```python
class SunSerialHost:
    def __init__(self, port: str, baudrate: int = 115200):
        ...

    def open(self) -> None:
        ...

    def close(self) -> None:
        ...

    def start_reader(self) -> None:
        ...

    def stop_reader(self) -> None:
        ...

    def send_command(self, frame: bytes) -> None:
        ...
```

接收线程职责：

1. 循环读取串口数据。
2. 把数据交给 `TelemetryParser.feed()`。

建议串口读取参数：


这样既不会长时间阻塞，也能承受 `10 Hz` 或更高输出频率。

## 21. 协议细节补充

### 21.1 为什么必须保留原始 ADC

太阳敏最终输出角度，但测试和标定不能只依赖角度。原因如下：

1. 如果角度误差大，需要判断问题来自光学结构、模拟链路、ADC、暗场偏置、通道增益，还是角度算法。
2. 四路 ADC 可以判断是否饱和、是否弱光、是否有通道异常。
3. 标定模型需要从原始特征量 `rx`、`ry` 建立到真实入射角的映射。
4. 后期即使固件算法改了，也可以用同一批 ADC 数据离线重算角度。

因此固件工程师必须在遥测帧中输出四路 ADC。若带宽担心不够，可以降低其他字段，但不要删 ADC。

### 21.2 上位机可计算的辅助量

即使 STM32 已经输出 `alpha`、`beta`，上位机仍建议实时计算以下辅助量：

```text
sx = adc_vax1 + adc_vax2
sy = adc_vay1 + adc_vay2
s_total = adc_vax1 + adc_vax2 + adc_vay1 + adc_vay2
rx = (adc_vax2 - adc_vax1) / (adc_vax2 + adc_vax1 + eps)
ry = (adc_vay2 - adc_vay1) / (adc_vay2 + adc_vay1 + eps)
```

这些量可以先不显示在主界面，但建议写入后处理脚本或调试面板。

如果暗场偏置已经由固件扣除，应在协议文档中说明。若未扣除，上位机后处理时可使用暗场数据进行扣除：

```text
ix1 = max(adc_vax1 - dark_vax1, 0)
ix2 = max(adc_vax2 - dark_vax2, 0)
iy1 = max(adc_vay1 - dark_vay1, 0)
iy2 = max(adc_vay2 - dark_vay2, 0)
```

### 21.3 CRC-16/Modbus 约定

推荐 CRC-16/Modbus 参数：

| 参数 | 值 |
|---|---|
| 初值 | `0xFFFF` |
| 多项式 | `0xA001` |
| 输入反转 | 是 |
| 输出反转 | 是 |
| 结果异或 | `0x0000` |
| 发送顺序 | 低字节在前 |

示例伪代码：

```text
crc = 0xFFFF
for byte in data:
    crc ^= byte
    repeat 8 times:
        if crc & 0x0001:
            crc = (crc >> 1) ^ 0xA001
        else:
            crc = crc >> 1
return crc & 0xFFFF
```

固件和上位机必须用同一个 CRC 输入范围。推荐 CRC 覆盖：

```text
version, msg_type, node_id, payload_len, payload
```

不覆盖 `sof`，不覆盖 `crc16` 本身。

### 21.4 帧序号与丢帧判断

`seq` 使用 `uint16`，范围 `0` 到 `65535`，溢出后回到 `0`。

丢帧计算：

```text
expected = (last_seq + 1) & 0xFFFF
if seq != expected:
    dropped += (seq - expected) & 0xFFFF
```

第一次收到帧时只记录 `last_seq`，不计算丢帧。

### 21.5 时间戳

建议同时记录两个时间：

| 时间 | 来源 | 用途 |
|---|---|---|
| `timestamp_ms` | STM32 | 分析下位机输出周期 |
| `pc_time_ms` | 上位机电脑 | 对齐测试操作、CSV 文件和人工记录 |

如果 STM32 复位，`timestamp_ms` 可能回到 `0`，上位机不应把它当作错误，只需正常记录。

## 24. 模拟数据源设计

因为 STM32 固件尚未完成，上位机应自带模拟数据源。模拟数据源可以让界面、协议、日志先跑起来。

模拟数据源至少支持三种模式：

| 模式 | 行为 | 用途 |
|---|---|---|
| 正常模式 | `10 Hz` 输出平滑变化角度和 ADC | 验证界面和 CSV |
| CRC 错误模式 | 每隔若干帧故意破坏 CRC | 验证错误统计 |
| 丢帧模式 | 帧序号周期性跳变 | 验证丢帧统计 |

模拟遥测建议：

```text
alpha = 30 * sin(t)
beta = 20 * cos(t)
adc_vax1 = 1800 - alpha * 10 + noise
adc_vax2 = 1800 + alpha * 10 + noise
adc_vay1 = 1800 - beta * 10 + noise
adc_vay2 = 1800 + beta * 10 + noise
temp = 25 + 2 * sin(t / 20)
```

模拟数据源可以放在 `sun_host.py` 或单独放在 `sun_simulator.py`。如果后续需要独立测试，建议单独建 `sun_simulator.py`。

## 25. 开发顺序建议

如果后续开始写代码，建议严格按以下顺序做：

1. 写 `sun_models.py`，先固定数据结构。
2. 写 `sun_protocol.py`，先用单元测试验证 CRC 和帧解析。
3. 写模拟数据源，生成合法二进制帧。
4. 写 `sun_logger.py`，验证 CSV 写入。
5. 写最小 `sun_gui.py`，只显示当前数值。
6. 接入 `sun_monitor.py` 曲线。
7. 接入真实串口 `sun_host.py`。
8. 做断线、超时、CRC 错误和丢帧处理。
9. 做标定采集区。
10. 最后再考虑遥控命令。

这个顺序的好处是，最难排查的串口硬件问题放在后面。前面先用模拟帧把协议、界面、CSV 全部验证好。

## 26. 单元测试建议

建议至少写以下测试：

| 测试名 | 输入 | 期望 |
|---|---|---|
| `test_crc16_modbus_known_vector` | 标准字节串 | CRC 等于已知值 |
| `test_parse_valid_telemetry_frame` | 合法遥测帧 | 得到正确字段 |
| `test_parser_handles_split_frame` | 一帧拆成两次 feed | 第二次返回遥测 |
| `test_parser_handles_sticky_frames` | 两帧粘在一起 | 一次返回两个遥测 |
| `test_parser_discards_noise_before_sof` | 前面带乱码 | 正常解析后续帧 |
| `test_parser_rejects_bad_crc` | CRC 错误帧 | 不返回遥测，错误计数加一 |
| `test_logger_writes_header_and_row` | 一帧遥测 | CSV 有表头和一行数据 |
| `test_drop_count_wraparound` | `65535 -> 0` | 不计丢帧 |

协议测试比 GUI 测试优先级更高。只要协议解析可靠，界面问题通常更容易定位。

## 27. 真实硬件联调步骤

### 27.1 通信链路检查

1. 给太阳敏供电，建议先使用限流电源。
2. USB-RS485 插入电脑，确认设备管理器中出现 COM 口。
3. 连接 `RS485_A/B` 和 GND。
4. 打开上位机，选择对应 COM 口。
5. 点击连接，观察是否有字节进入。
6. 如果无数据，用串口助手交叉验证。
7. 如果乱码或 CRC 错误高，尝试交换 A/B。
8. 如果仍异常，检查 STM32 波特率、数据位、停止位、DE 控制。

### 27.2 ADC 检查

1. 暗场遮光，记录四路 ADC。
2. 用稳定光源照射，观察四路 ADC 是否上升。
3. 遮挡不同方向，观察对应通道是否变化。
4. 检查是否有通道固定为 `0` 或满量程。
5. 检查强光下是否饱和。

如果某一路始终不变，优先检查硬件焊接、STM32 ADC 通道配置和固件数组顺序。

### 27.3 角度方向检查

1. 转台置 `alpha=0, beta=0`，记录中心输出。
2. 转到 `alpha=+10, beta=0`，观察 `alpha_deg` 是否增大。
3. 转到 `alpha=-10, beta=0`，观察 `alpha_deg` 是否减小。
4. 转到 `alpha=0, beta=+10`，观察 `beta_deg` 是否增大。
5. 转到 `alpha=0, beta=-10`，观察 `beta_deg` 是否减小。

如果符号相反，不一定是硬件错，也可能是坐标系定义相反。应在 ICD 和报告中固定符号约定。

### 27.4 标定采集流程

1. 启动太阳模拟器并预热。
2. 将太阳敏安装到转台夹具。
3. 连接供电和 RS-485。
4. 上位机连接串口。
5. 输入测试点编号 `P001`，参考角 `0,0`。
6. 点击开始记录，采集 `5 s` 到 `10 s`。
7. 停止记录或切换测试点。
8. 按网格扫描 `alpha_ref` 和 `beta_ref`。
9. 每个点位记录同样时长。
10. 测试结束后保存 CSV 并备份。

建议第一轮网格不要太密，先做：

```text
(0, 0)
(+10, 0), (-10, 0), (0, +10), (0, -10)
(+30, 0), (-30, 0), (0, +30), (0, -30)
(+60, 0), (-60, 0), (0, +60), (0, -60)
```

确认链路正常后再做完整二维网格。

## 28. 与技术报告的对应关系

上位机输出的数据可以直接支撑技术报告中的以下内容：

| 技术报告内容 | 上位机数据来源 |
|---|---|
| 通信接口验证 | 串口连接状态、帧率、CRC 错误 |
| 太阳存在检测 | `sun_present`、`signal_sum` |
| 入射角输出 | `alpha_deg`、`beta_deg` |
| 温度输出 | `temp_c` |
| 状态字 | `status_word` 和状态位解析 |
| 全视场精度 | `alpha_ref/beta_ref` 与 `alpha/beta` |
| 重复性 | 同一 `test_point` 多次记录 |
| 稳定性 | 固定点长时间 CSV |
| 饱和测试 | `saturation_flag` 和 ADC 满量程 |
| 暗场测试 | 遮光条件下四路 ADC |

因此 CSV 字段必须从第一版开始保持稳定。后续即使界面变化，也不要随意改 CSV 列名。

## 29. 需要尽快和固件工程师确认的问题

以下问题会直接影响上位机代码：

1. 固件是主动周期上报，还是等待上位机查询。
2. 默认输出频率是 `5 Hz`、`10 Hz` 还是更高。
3. 是否同意使用本文推荐的 `32` 字节二进制遥测帧。
4. ADC 是 `12 bit` 原始值还是已经换算为毫伏。
5. ADC 是否已经做平均滤波。
6. ADC 是否已经扣除暗场偏置。
7. `alpha` 和 `beta` 是固件标定后的角度，还是未标定几何角。
8. 角度单位是否接受 `0.01 deg` 的 `int16` 表示。
9. 温度来自 STM32 内部温度还是外部温度传感器。
10. `status_word` 每一位如何定义。
11. 是否有固件版本号字段。
12. 是否支持节点 ID 配置。
13. 是否需要上位机发送遥控命令。
14. 半双工 RS-485 的 `DE` 控制时序如何实现。
15. 上电后多长时间开始输出遥测。

建议把这些问题整理成一页 `太阳敏通信协议ICD.md`，让固件工程师确认后再开始最终联调。

## 31. 后续代码实现时的注意事项

1. 不要把协议解析、串口读取和 UI 更新写在一个大文件里。
2. 不要在 UI 主线程里阻塞读取串口。
3. 不要假设一次串口读取就是一帧。
4. 不要只记录角度，必须记录四路 ADC。
5. 不要用中文字段名做 CSV 表头，后续 Python 分析不方便。
6. 不要让 CSV 列名随版本频繁变化。
7. 不要在没有 CRC 的情况下做正式测试记录。
8. 不要在固件协议未定时做太复杂的遥控功能。
9. 不要把上位机做成只适合演示的界面，工程测试数据更重要。
10. 不要忽略异常处理，现场测试最常见的问题就是串口断开、A/B 接反、数据超时和协议不一致。

## 32. 建议的下一步

下一步建议先写 `太阳敏通信协议ICD.md`，把本文第 7、8、21、29 节中涉及固件的内容压缩成一份工程师可确认的接口文档。ICD 确认后，再开始上位机代码实现。

在固件未完成前，上位机可以先按本文推荐协议实现模拟数据源。这样等工程师给出 STM32 固件后，只需要调整协议字段和 CRC 范围，不需要重写界面和日志系统。
